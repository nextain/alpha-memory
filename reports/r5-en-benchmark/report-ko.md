## 1. 프로젝트 소개

### AI는 왜 기억이 필요한가?

사람과 오래 대화한 AI일수록 더 도움이 됩니다. 처음 만나는 사람에게 직업, 취미, 가족 이야기를 매번 반복해야 한다면 피곤하듯, AI도 사용자를 기억하지 못하면 진정한 조력자가 될 수 없습니다.

하지만 단순히 "기억을 많이 하는 것"이 답이 아닙니다. 기억을 너무 많이 하면 엉뚱한 정보를 꺼내거나, 없는 사실을 만들어내는 "환각(hallucination)" 문제가 생깁니다. **Alpha Memory**는 이 문제를 해결하기 위해 인간의 인지과학에서 영감을 받은 AI 메모리 시스템입니다.

### Alpha Memory 소개

Alpha Memory는 **Naia OS**(넥스테인의 오픈소스 AI 데스크탑 OS) 프로젝트의 핵심 메모리 패키지입니다. 인간의 뇌를 모방한 4개의 기억 저장소로 구성됩니다:

| 저장소 | 뇌 유사 부위 | 무엇을 저장하나 |
|--------|------------|----------------|
| **에피소딕(Episodic)** | 해마 | 시간 순서가 있는 경험과 사건 |
| **시맨틱(Semantic)** | 대뇌피질 | 사실, 개념, 관계 |
| **절차적(Procedural)** | 기저핵 | 기술, 전략, 학습된 패턴 |
| **작업(Working)** | 전전두엽 | 현재 대화의 활성 컨텍스트 |

핵심 기술 요소:
- **중요도 가중 기억 (3축 점수)**: 중요도 × 놀라움 × 감정으로 기억할 가치를 판단
- **에빙하우스 망각 곡선**: 시간이 지날수록 중요도가 낮아지는 자연스러운 망각 구현
- **지식 그래프**: 사람·장소·관계를 그래프로 연결하여 문맥 검색
- **모순 감지**: 새 정보가 기존 기억과 충돌할 때 자동으로 감지·해결

## 2. 테스트 설계

### 어떻게 테스트했나?

**기본 설정:**
- 언어: 영어
- 페르소나: 가상의 인물에 대한 1,000개의 사실(직업, 취미, 가족, 과거 경험 등)이 담긴 "사실 은행"
- 총 테스트 항목: 240개 질의
- 채점 방식: GLM-5.1 AI가 "의미 기반" 채점 (단순 키워드 일치가 아닌 의미 파악)
- 응답 모델: Gemini 2.5 Flash Lite (모든 시스템 동일 조건)

**비교 대상 9개 시스템:**

| 시스템 | 설명 |
|--------|------|
| **naia** | Alpha Memory (본 프로젝트) — 4-store 인지 아키텍처 |
| **letta** | Letta (구 MemGPT) — 에이전트 메모리 관리 프레임워크 |
| **mem0** | mem0 OSS — LLM 앱을 위한 오픈소스 메모리 레이어 |
| **open-llm-vtuber** | Open-LLM-VTuber — VTuber 스트리밍 AI 캐릭터 메모리 |
| **sillytavern** | SillyTavern — 롤플레이 챗봇 메모리 |
| **sap** | SAP AI Core 메모리 컴포넌트 |
| **graphiti** | Graphiti (getzep) — Neo4j 시간 지식 그래프 |
| **openclaw** | OpenClaw — 경량 메모리 레이어 |
| **airi** | AIRI (moeru-ai) — **메모리 없음 베이스라인** |

> airi는 메모리 시스템이 전혀 없는 순수 LLM으로, 다른 시스템과의 비교 기준선(baseline)으로 포함했습니다.

### 채점 기준

- **핵심 테스트**: 가중치 기반 통과율로 점수 산출
- **등급**: A (90%+), B (75%+), C (60%+), F (<60%)
- **특별 실격**: abstention(환각 방지) 테스트에서 하나라도 실패하면 다른 점수와 무관하게 **F (abstention fail)**

### 이번 벤치마크에서 측정하지 않은 것: 속도와 비용

이번 R5 벤치마크는 **정확도만 측정**했습니다. 실제 AI 시스템 선택에서 동등하게 중요한 두 축 — 속도(응답 지연)와 비용(토큰/인프라 비용) — 은 이번 라운드에서 제외되었습니다. 이는 단순한 측정 누락이 아니라, Alpha Memory의 개선 방향을 올바르게 잡기 위해 반드시 보완해야 할 기준입니다.

**속도 (응답 지연)**

메모리 아키텍처에 따라 응답 지연은 수십 배까지 차이가 납니다.

| 아키텍처 | 검색 방식 | 추정 지연 | 비고 |
|---------|---------|---------|----|
| 벡터 검색 (naia, mem0) | HNSW 근사 최근접 | 낮음 (~수 ms) | 임베딩 API 호출 시 외부 지연 가산 |
| 키워드/문자열 (SillyTavern) | SQLite 전문 검색 | 매우 낮음 | 의미 검색 불가 |
| LLM 기반 검색 (letta) | 검색에 LLM 호출 | 높음 (+수 초) | 검색 자체가 LLM 추론 |
| 그래프 순회 (graphiti) | Neo4j Cypher 쿼리 | 가변적 | 홉(hop) 수에 따라 O(n²) 가능 |

> 이번 벤치마크의 응답 LLM은 모든 시스템 동일(Gemini 2.5 Flash Lite). 따라서 시스템 간 속도 차이는 **메모리 검색 아키텍처 차이**에서만 발생한다.

**비용 (토큰/인프라)**

메모리 방식에 따라 쿼리당 비용 구조가 근본적으로 다릅니다.

| 아키텍처 | 저장 비용 | 검색 비용 | 특이사항 |
|---------|---------|---------|--------|
| 벡터 임베딩 (naia, mem0) | 임베딩 API 단가 × 사실 수 | 임베딩 1회 | naia는 4-store 각각 임베딩 → 4배 |
| 컨텍스트 삽입 (SillyTavern) | 무료(로컬 저장) | 프롬프트 토큰 직접 소모 | 기억이 많을수록 입력 토큰 급증 |
| LLM 검색 (letta) | 임베딩 1회 | LLM 추가 호출 비용 | 응답당 LLM 2회 소모 |
| 그래프 (graphiti) | Neo4j 인프라 비용 | 쿼리 비용(LLM 무관) | Neo4j 운영 인프라 필요 |

**Naia에 대한 함의**

현재 naia의 비용 구조는 4-store 아키텍처로 인해 **단일 사실 저장 시 최대 4회 임베딩 API 호출**이 발생합니다. Gemini `text-embedding-004` 기준 768차원 벡터를 4개 저장하면 단순 산술로 mem0 대비 4배 비용입니다. 실용화 전 다음이 반드시 선행되어야 합니다:
- 어떤 사실을 어느 store에 넣을지 **선택적 라우팅** (모든 사실을 4-store 전부에 넣지 않기)
- 에빙하우스 망각 기반 **배치 압축** — 접근 빈도 낮은 기억은 1개 store로 통합
- 검색 지연 측정을 포함한 **다음 벤치마크 라운드** 설계

## 3. 12가지 평가 카테고리 상세 설명

### 🔵 핵심 메모리 능력

**1. direct_recall (직접 회상)** — 25문항
> "저장한 사실을 직접 꺼낼 수 있는가?"

예시: 사용자가 "나는 소프트웨어 엔지니어야"라고 알려준 후, "내 직업이 뭐야?"라고 물을 때 정확히 답하는지 테스트합니다.

**2. semantic_search (의미 기반 검색)** — 25문항
> "키워드가 아닌 의미로 관련 기억을 찾을 수 있는가?"

예시: "내 취미에 대해 얘기해줘"라는 질문에 '독서', '등산' 등 취미 관련 기억들을 연결해 답하는지 봅니다. 단순 키워드 검색으로는 불가능하고, 의미적 연결이 필요합니다.

**3. proactive_recall (선제적 회상)** — 20문항
> "묻지 않아도 먼저 관련 정보를 제안하는가?"

예시: 사용자가 "서울 여행 계획 중이야"라고 말할 때, AI가 자발적으로 "아, 서울에 친구가 있다고 하셨죠?"라고 꺼내는 것. 수동적 응답이 아닌 능동적 제안 능력을 측정합니다.

**4. abstention (환각 방지)** — 20문항
> "모르는 것은 모른다고 말하는가?"

예시: 사용자가 혈액형을 알려준 적 없는데 "내 혈액형이 뭐야?"라고 물을 때. AI가 없는 정보를 지어내면 FAIL, "그런 정보가 없어요"라고 하면 PASS. 이 테스트를 하나라도 실패하면 전체 등급이 F입니다.

### 🟡 고급 메모리 능력

**5. irrelevant_isolation (개인정보 격리)** — 15문항
> "관련 없는 질문에 개인정보를 끼워 넣지 않는가?"

예시: "파이썬 리스트 정렬하는 방법은?"이라고 물을 때, AI가 "안녕하세요 홍길동님, 소프트웨어 엔지니어이시죠? 정렬 방법은..."처럼 불필요하게 개인정보를 포함하면 FAIL입니다.

**6. multi_fact_synthesis (다중 사실 통합)** — 20문항
> "여러 기억을 조합해 복합적인 답을 만들 수 있는가?"

예시: "내 예산, 취미, 일정을 고려해서 다음 주 활동을 추천해줘" → 직업·소득·취미·시간대 등 여러 사실을 동시에 연결해야 합니다.

**7. entity_disambiguation (동명이인 구분)** — 20문항
> "같은 이름의 다른 사람을 맥락으로 구분하는가?"

예시: 친구 "Alex(개발자)"와 형 "Alex(의사)"가 저장된 상황에서 "Alex한테 물어봤어"라고 했을 때, 어떤 Alex를 말하는지 맥락으로 파악하는 능력입니다.

**8. contradiction_direct (직접 모순 처리)** — 20문항
> "명확한 정보 변경을 정확히 업데이트하는가?"

예시: "나 이제 카페 알바 그만뒀어"라고 했을 때, 기존의 "카페 알바 중" 기억을 적절히 업데이트하는지 봅니다.

**9. contradiction_indirect (간접 모순 처리)** — 15문항
> "암시적인 모순을 추론으로 감지하는가?"

예시: "담배 끊은 지 3년 됐어"라고 했을 때, 이전에 저장된 "흡연자" 기억이 자동으로 업데이트되어야 합니다. 직접적인 수정 명령 없이 맥락으로 추론해야 합니다.

**10. noise_resilience (노이즈 저항성)** — 20문항
> "관련 없는 정보들 사이에서 올바른 기억을 찾는가?"

예시: 대화 중 여러 무관한 주제가 섞여 있을 때, 정작 필요한 기억을 정확히 꺼내는 능력입니다.

### 🔴 지속성·시간 능력

**11. unchanged_persistence (불변 기억 보존)** — 15문항
> "일부 정보 업데이트 후 다른 기억들은 그대로인가?"

예시: 직업을 업데이트한 후 "내 이름이 뭐야?" "취미는?"처럼 변경하지 않은 다른 사실들도 여전히 정확히 기억하는지 봅니다.

**12. temporal (시간적 기억)** — 25문항
> "현재 상태와 과거 상태를 구분하여 회상할 수 있는가?"

예시: "내가 예전에 무슨 일 했었어?"라고 물을 때, 현재 직업이 바뀌었더라도 과거의 직업 기록을 정확히 꺼내는 능력입니다.

## 5. 세 AI의 분석

---

### [Claude Sonnet의 분석]

R5 결과는 **기억력과 자기인식(self-awareness)의 근본적 트레이드오프**를 정량적으로 드러낸다.

**기술적 분석**: Naia는 3위(84.0%)로 준수하나, abstention(45%)과 unchanged_persistence(47%)의 저조가 아킬레스건이다. 역설적으로 이 두 항목 실패는 같은 뿌리에서 발생한다 — 메모리가 풍부할수록 모르는 것도 "아는 척"하고, 업데이트 후 이전 기억과의 경계가 흐려진다. Graphiti는 Neo4j 기반 지식 그래프의 한계를 보여준다: contradiction 100%라는 구조적 강점이 있지만 semantic_search 4%는 그래프 탐색이 임베딩 기반 유사도 검색을 대체할 수 없음을 방증한다.

**설계 통찰**: Letta가 1위를 차지한 이유는 단일 지표 최적화가 아닌 균형에 있다. direct_recall 96%와 proactive_recall 100%의 공존은 검색(reactive)과 추천(proactive) 경로를 분리 설계했음을 시사한다. 메모리 시스템의 성숙도는 "얼마나 많이 기억하는가"가 아니라 **"언제 모른다고 말하는가"** 로 측정된다.

**Naia 개선 방향**:
1. **Confidence-gated recall**: 검색 유사도 임계값 미달 시 응답 억제 로직 추가 (abstention 복원)
2. **Immutable snapshot store**: 업데이트 전 구 사실을 `superseded` 태그로 보존, temporal/unchanged_persistence 동시 해결
3. **Dual-path retrieval**: 키워드 경로(direct_recall) + 임베딩 경로(semantic) 병렬 실행 후 교차 검증

**독창적 관찰**:
- airi(no-memory)의 abstention 100%는 블랙유머다 — 기억이 없으면 환각도 없다. 이는 메모리 시스템이 해결해야 할 문제가 "저장"이 아닌 **"경계 인식"** 임을 역설한다.
- 메모리 있는 모든 시스템의 abstention 동반 하락(40~65%)은 개별 구현 버그가 아닌 **메모리-확신 간 구조적 결합** 문제다. 별도의 "불확실성 모델링" 레이어가 업계 표준이 되어야 할 시점이다.
- Naia의 noise_resilience 100%는 4-store 아키텍처의 잠재력을 보여준다. 복잡한 인지 구조가 노이즈 필터링에 실질적으로 기여하고 있다.
- SillyTavern의 temporal 96%는 설계 원칙의 역설을 보여준다. 롤플레이 챗봇용으로 설계된 시스템이 temporal에서 1위를 차지한 것은, "캐릭터의 과거 발언과 현재 상태의 연속성"을 유지하는 것이 롤플레이의 핵심이기 때문이다. 목적이 명확한 시스템이 범용 메모리보다 특정 능력에서 우월할 수 있다.
- multi_fact_synthesis 100%와 abstention 저조는 서로 반대 방향의 능력이다 — 하나는 "연결할 수 있는 것을 연결하라", 다른 하나는 "없는 것을 만들지 말라". 이 둘을 모두 만족하는 시스템은 아직 존재하지 않는다. 이것이 AI 메모리 시스템의 본질적 이진 딜레마일 것이다.

*가장 중요한 단일 시사점: 메모리 시스템을 "더 많이 기억하도록" 최적화하는 것은 틀린 방향이다. 진짜 과제는 무엇을 기억하고, 무엇을 잊고, 무엇을 모른다고 말해야 하는지를 구분하는 능력이다.*

- **naia (84%)**: 아직 미완성이지만 잠재력이 가장 큰 시스템. noise_resilience와 contradiction_indirect 100%는 "흔들리지 않는" 기억의 안정성을 보여줍니다. 하지만 abstention 45%는 "너무 많이 알아서 모른다고 말하기 어려운" 역설적 상황을 연출합니다.
- **graphiti (55.8%)**: 완전히 다른 철학의 시스템. 기억의 "일관성"을 완벽하게 유지하지만 자연어로 꺼내는 능력이 없습니다. 사용자 경험보다 데이터 무결성에 최적화되어 있습니다.
- **airi (33.9%, no-memory)**: 아이러니하게도 abstention 100%. 아무것도 모르니 틀릴 것도 없습니다. 이는 "모른다"는 말이 지능의 척도가 아님을 보여줍니다.

**생태계 관점 — 어떤 시스템이 어디에 맞나?**

- 능동적 AI 비서·컴패니언 앱: **letta** (proactive_recall, 균형)
- AI 롤플레이·창작 캐릭터: **sillytavern** (temporal 96%, 기억 지속성 100%)
- 장기 지식 관리 플랫폼: **naia** (multi_fact_synthesis, contradiction_indirect 모두 100%)
- 기업 데이터 무결성 시스템: **graphiti** (contradiction 완벽, 데이터 일관성 보장)

**"환각 방지의 역설" — AI 신뢰도의 본질**

이번 벤치마크에서 가장 흥미로운 발견은 "환각 방지의 역설"입니다. 메모리가 강한 시스템일수록 abstention 점수가 낮습니다. letta 50%, naia 45%, mem0 40%... 반면 메모리 없는 airi는 100%입니다.

이것이 의미하는 바: 풍부한 기억은 자신감을 높이고, 과도한 자신감은 환각을 낳습니다. AI의 신뢰성은 "많이 아는 것"이 아니라 "자신이 모른다는 사실을 아는 것(메타인지)"에 달려 있습니다. 미래의 AI 메모리 시스템은 회상 능력뿐 아니라 **불확실성을 명시적으로 표현하는 능력**을 함께 갖춰야 합니다.

**독창적 관점**:
- 이 벤치마크는 "회상" 능력을 측정하지만, 진정으로 유용한 기억 시스템은 **성찰(reflection)** 능력 — 과거의 실수를 학습하는 능력 — 도 갖춰야 합니다. 다음 벤치마크에는 이 차원이 추가되어야 합니다.
- "기억의 비용"이 측정되지 않았습니다. 완벽한 기억은 공짜가 아닙니다 — 토큰 비용, 응답 지연, 저장 비용. 실용적 AI 시스템은 효율적인 망각을 통해 중요한 정보만 선별 유지해야 합니다.
- unchanged_persistence에서 airi가 100%를 기록한 것은 "변경한 것이 없으니 망가진 것도 없다"는 의미입니다. 이는 정적 시스템이 동적 시스템보다 특정 지표에서 유리한 구조적 편향입니다.

### [GLM-5.1의 분석]

**1. 구현 중심 아키텍처 분석**

각 시스템의 아키텍처 선택은 벤치마크 결과에 뚜렷한 차이를 보입니다. **letta**(87.5%)와 **open-llm-vtuber**(85.2%)는 에이전트 컨텍스트 관리에 최적화되어 direct_recall(96%)과 proactive_recall(100%)에서 압도적인 성능을 보입니다. 반면 **naia**(84.0%)는 4-store 구조 덕분에 noise_resilience(100%)와 contradiction_indirect(100%)를 달성했으나, 사실 업데이트 시 다른 기억이 삭제되는 치명적인 버그(naia-os#221)로 인해 unchanged_persistence(47%)가 급락했습니다. **sillytavern**(79.8%)은 롤플레이 특성상 temporal(96%)에는 강하지만, 능동적 회상(60%)에서 부진합니다.

**3. 실용적 권장: AI 컴패니언 앱을 만든다면**

1위인 **letta**를 선택하겠습니다. 컴패니언의 핵심은 사용자의 질문 없이도 기억을 꺼내는 proactive_recall(100%)과 정확한 회상(96%)입니다. abstention(50%)으로 환각 방지가 취약하지만, 컴패니언 앱에서는 아무것도 기억 못하는 것이 잘못 기억하는 것보다 더 치명적입니다.

**4. 벤치마크 설계에 대한 비판적 시각**

1. **abstention의 역설**: 메모리 없는 airi/openclaw가 abstention 100%를 받지만, 이는 '아는 것이 없어서 모른다'는 응답입니다. 벤치마크가 이 경우를 '좋은 것'으로 채점하는 것은 측정 지표의 한계입니다.
2. **정적 테스트의 한계**: 1,000개 팩트 뱅크 기반이므로 실제 환경의 토큰 비용·응답 지연·확장성을 측정하지 못합니다.
3. **다국어 편향**: 영어 기반 테스트이므로 한국어, 일본어 등 다른 언어 환경에서의 성능 차이는 알 수 없습니다.
4. **사용자 정의 적응성 미측정**: 각 사용자의 고유한 말투·표현 패턴에 적응하는 능력이 포함되지 않았습니다.

## 6. 세 AI의 토론: 합의 사항

세 AI가 독립적으로 분석한 후 핵심 논점에 대해 의견을 교환했습니다.

### 합의 1: "abstention 역설"은 메모리 업계 전체의 미해결 과제

**Claude**: 메모리-확신의 구조적 결합 문제다. 불확실성 모델링 레이어가 별도로 필요하다.

**Gemini**: 사용자 신뢰 관점에서 "모른다"고 말하는 것이 AI 성숙도의 진짜 척도다.

**GLM**: 벤치마크 설계 자체가 이 문제를 공정하게 측정하지 못하고 있다. airi의 100%는 "무능한 성공"이다.

**합의**: 고성능 메모리 시스템이 abstention에서 낮은 점수를 받는 것은 구현 버그가 아닌 구조적 긴장 관계다. 다음 세대 메모리 시스템은 **"회상 자신감 점수"** 를 별도로 관리하는 레이어가 필요하다.

### 합의 2: graphiti는 실패한 게 아니라 다른 게임을 하고 있다

**Claude**: semantic_search 4%는 그래프 탐색과 임베딩 검색이 근본적으로 다른 기술임을 보여준다.

**Gemini**: 데이터 무결성·기업 환경에서는 graphiti가 오히려 적합하다. 다른 평가 기준이 필요한 시스템이다.

**GLM**: 그래프 DB와 벡터 DB는 상호 보완 관계다. 두 기술을 합친 하이브리드 아키텍처가 답이다.

**합의**: 단일 지표로 메모리 시스템을 비교하는 것에는 한계가 있다. "활용 목적별" 최적 시스템이 다르며, 이 벤치마크는 "AI 컴패니언" 사용 사례에 편향되어 있음을 인정해야 한다.

### 합의 3: Naia의 현 위치와 가야 할 방향

**Claude**: 84%는 좋은 시작이지만 abstention(45%)과 unchanged_persistence(47%)의 동반 해결 없이는 실용화가 어렵다.

**Gemini**: noise_resilience·multi_fact_synthesis 100%는 경쟁력 있는 차별점이다. 이 강점을 유지하면서 약점을 보완해야 한다.

**GLM**: unchanged_persistence는 능력 한계가 아닌 버그(naia-os#221)다. 수정 후 재측정이 필요하다.

**합의**: Naia는 현재 "높은 잠재력을 가진 불완전한 시스템"이다. 1개의 버그 수정(#221)만으로도 순위가 크게 달라질 가능성이 있다.

## 7. Naia 개선 방향 상세 로드맵

### 단기 (1-2스프린트)

**[P0] naia-os#221 — unchanged_persistence 버그 수정**
- 원인: contradiction resolution 시 연관 엔티티 cascade 덮어쓰기
- 수정: 엔티티 단위 트랜잭션 격리, 수정 대상 속성만 업데이트
- 예상 효과: unchanged_persistence 47% → 85%+ 전망

**[P1] Confidence-gated recall — abstention 개선**
- 검색 코사인 유사도 0.75 미만 시 "해당 정보 없음" 명시 반환
- 예상 효과: abstention 45% → 70%+ 전망

### 중기 (1분기)

**[P2] Bi-temporal 모델 도입 — temporal 개선**
- 각 사실에 `valid_from`, `valid_until` 타임스탬프 추가
- 과거 상태 쿼리 가능: "내가 2년 전에 무슨 일 했었어?"
- 예상 효과: temporal 80% → 90%+ 전망

**[P3] Dual-path retrieval — direct_recall/semantic 개선**
- 키워드 BM25 경로 + 임베딩 경로 병렬 실행 후 RRF(Reciprocal Rank Fusion) 통합
- 예상 효과: direct_recall 84% → 92%+ 전망

### 장기 (2026 H2)

**[P4] 불확실성 명시 레이어**
- 모든 recall 응답에 신뢰도 점수 포함
- LLM 프롬프트에 "신뢰도 X% 미만 정보는 추측임을 명시" 지시 추가

**[P5] 메모리 효율 최적화**
- 에빙하우스 망각 곡선 기반 자동 압축·아카이빙
- 토큰 비용 50% 절감 목표
