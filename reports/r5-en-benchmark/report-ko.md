## 1. 프로젝트 소개

### AI는 왜 기억이 필요한가?

사람과 오래 대화한 AI일수록 더 도움이 됩니다. 처음 만나는 사람에게 직업, 취미, 가족 이야기를 매번 반복해야 한다면 피곤하듯, AI도 사용자를 기억하지 못하면 진정한 조력자가 될 수 없습니다.

하지만 단순히 "기억을 많이 하는 것"이 답이 아닙니다. 기억을 너무 많이 하면 엉뚱한 정보를 꺼내거나, 없는 사실을 만들어내는 "환각(hallucination)" 문제가 생깁니다. **Alpha Memory**는 이 문제를 해결하기 위해 인간의 인지과학에서 영감을 받은 AI 메모리 시스템입니다.

### Alpha Memory 소개

Alpha Memory는 **Naia OS**(넥스테인의 오픈소스 AI 데스크탑 OS) 프로젝트의 핵심 메모리 패키지입니다. 인간의 뇌를 모방한 4개의 기억 저장소로 구성됩니다:

| 저장소 | 뇌 유사 부위 | 무엇을 저장하나 |
|--------|------------|----------------|
| **에피소딕(Episodic)** | 해마 | 시간 순서가 있는 경험과 사건 |
| **시맨틱(Semantic)** | 대뇌피질 | 사실, 개념, 관계 |
| **절차적(Procedural)** | 기저핵 | 기술, 전략, 학습된 패턴 |
| **작업(Working)** | 전전두엽 | 현재 대화의 활성 컨텍스트 |

핵심 기술 요소:
- **중요도 가중 기억 (3축 점수)**: 중요도 × 놀라움 × 감정으로 기억할 가치를 판단
- **에빙하우스 망각 곡선**: 시간이 지날수록 중요도가 낮아지는 자연스러운 망각 구현
- **지식 그래프**: 사람·장소·관계를 그래프로 연결하여 문맥 검색
- **모순 감지**: 새 정보가 기존 기억과 충돌할 때 자동으로 감지·해결

## 2. 테스트 설계

### 어떻게 테스트했나?

**기본 설정:**
- 언어: 영어
- 페르소나: 가상의 인물에 대한 1,000개의 사실(직업, 취미, 가족, 과거 경험 등)이 담긴 "사실 은행"
- 총 테스트 항목: 240개 질의
- 채점 방식: GLM-5.1 AI가 "의미 기반" 채점 (단순 키워드 일치가 아닌 의미 파악)
- 응답 모델: Gemini 2.5 Flash Lite (모든 시스템 동일 조건)

**비교 대상 9개 시스템:**

| 시스템 | 설명 |
|--------|------|
| **naia** | Alpha Memory (본 프로젝트) — 4-store 인지 아키텍처 |
| **letta** | Letta (구 MemGPT) — 에이전트 메모리 관리 프레임워크 |
| **mem0** | mem0 OSS — LLM 앱을 위한 오픈소스 메모리 레이어 |
| **open-llm-vtuber** | Open-LLM-VTuber — VTuber 스트리밍 AI 캐릭터 메모리 |
| **sillytavern** | SillyTavern — 롤플레이 챗봇 메모리 |
| **sap** | SAP AI Core 메모리 컴포넌트 |
| **graphiti** | Graphiti (getzep) — Neo4j 시간 지식 그래프 |
| **openclaw** | OpenClaw — 경량 메모리 레이어 |
| **airi** | AIRI (moeru-ai) — **메모리 없음 베이스라인** |

> airi는 메모리 시스템이 전혀 없는 순수 LLM으로, 다른 시스템과의 비교 기준선(baseline)으로 포함했습니다.

### 채점 기준

- **핵심 테스트**: 가중치 기반 통과율로 점수 산출
- **등급**: A (90%+), B (75%+), C (60%+), F (<60%)
- **특별 실격**: abstention(환각 방지) 테스트에서 하나라도 실패하면 다른 점수와 무관하게 **F (abstention fail)**

### 속도와 비용: 아키텍처 분석 기반 추정

R5는 정확도만 측정했습니다. 그러나 **naia-os에 Alpha Memory를 탑재하는 것이 무엇을 의미하는지** 평가하려면 속도와 비용을 따지고 시스템을 선택할 수 없습니다. 직접 타이밍은 하지 않았지만, **어댑터 코드 분석**과 **벤치마크에서 실제 수집된 콘텍스트 크기 데이터**를 기반으로 추정이 가능합니다.

**전제: 백그라운드 메모리 모듈**

naia-os에서 메모리 모듈은 두 가지 진입점으로 작동합니다:
- **addFact (기억 저장)**: 사용자 발화 후 백그라운드 비동기 실행 → 응답 속도 직접 영향 없음. **운영 비용에 직결**
- **search (기억 검색)**: 응답 생성 전 동기 실행 → **사용자가 직접 체감하는 대기 시간**

**검색 지연 추정 (UX 직결)**

어댑터 코드 분석 기반 추정치:

| 시스템 | 검색 방식 | 추정 지연 |
|--------|---------|----------|
| graphiti | Neo4j Cypher만 (LLM 미사용) | ~50ms |
| naia | Gemini embed API + 로컈 HNSW | ~100ms |
| mem0 | Gemini embed API + SQLite 벡터 | ~105ms |
| openclaw | Gemini embed API + FTS5 하이브리드 | ~105ms |
| sap | Gemini embed API + FAISS + BM25 | ~110ms |
| letta | Gemini embed API + Letta 서버 왕복 | ~300ms |
| sillytavern | 로컈 transformers.js + vectra | ~500ms |
| open-llm-vtuber | LLM recall 호삶 + embed + 검색 | ~2,500ms |
| airi | 없음 (no-memory) | 0ms |

> 모든 시스템의 응답 LLM은 Gemini 2.5 Flash Lite로 동일. 시스템 간 지연 차이는 메모리 검색 레이어에서만 발생한다.

**비용 구조 (addFact API + 콘텍스트 토큰)**

실제 벤치마크에서 관측된 콘텍스트 크기 실측치를 함께 표시:

| 시스템 | addFact API 호출 | 콘텍스트 크기 (실측 평균) | LLM 입력 토큰 영향 |
|--------|------------|------------------|--------------|
| airi | 없음 | 0자 | 없음 |
| graphiti | LLM 1회 + Neo4j 쓰기 | 166자 | 낙음 |
| open-llm-vtuber | LLM **2회** + embed 1회 | 242자 | 높음 (LLM recall 포함) |
| letta | LLM 1회 + embed 1회 | 259자 | 높음 |
| naia | embed 1회 | 320자 | 중간 |
| mem0 | LLM 1회 + embed 1회 | 320자 | 중간 |
| sap | LLM 1회 + embed 1회 | 334자 | 중간 |
| sillytavern | 로컈 embed (무료) | 527자 | 없음(로컈, 컨텍스트 크지만) |
| openclaw | embed 1회 | 0자 (검색 실패) | 없음 |

> 콘텍스트 크기는 이번 벤치마크에서 각 어댑터가 LLM에 실제 전달ํ 기억 콘텍스트의 평균 문자 수(실측)입니다. 크면 클수록 에토 LLM 입력 토큰 비용이 증가합니다.

**naia-os 탑재 관점에서의 평가**

이 데이터는 naia 개선에 대해 명확한 방향을 제시합니다:

| 항목 | 평가 | 개선 방향 |
|------|------|----------|
| 검색 지연 | 양호(~100ms) | 로컈 임베딩 백엔드 교체 시 0코 지연 가능 |
| addFact LLM 호출 | **없음 (우위)** | letta/mem0/sap/olv는 LLM 1-2회, naia는 embed만 → addFact 비용 최소 |
| 콘텍스트 실승 | 적정(320자) | graphiti(166자)에 비해 높지만, sillytavern(527자)에 비해 효율적 |
| embed 백엔드 | Gemini API(외부) | **로컈 임베딩 옵션 미지원** — sillytavern은 무료 로컈 동작. naia-os 사용자는 API 키 필수 |

**핵심 결론**: naia는 addFact 비용에서 최우수이고(LLM 호출 없음), 검색 지연도 양호하다. 단, **embed API가 외부 의존이라는 구조적 문제**는 로컬 임베딩 모델 지원으로 해결할 수 있으며, 이는 naia-os 로드맵과 일치하는 실행 과제다.

## 3. 12가지 평가 카테고리 상세 설명

### 🔵 핵심 메모리 능력

**1. direct_recall (직접 회상)** — 25문항
> "저장한 사실을 직접 꺼낼 수 있는가?"

예시: 사용자가 "나는 소프트웨어 엔지니어야"라고 알려준 후, "내 직업이 뭐야?"라고 물을 때 정확히 답하는지 테스트합니다.

**2. semantic_search (의미 기반 검색)** — 25문항
> "키워드가 아닌 의미로 관련 기억을 찾을 수 있는가?"

예시: "내 취미에 대해 얘기해줘"라는 질문에 '독서', '등산' 등 취미 관련 기억들을 연결해 답하는지 봅니다. 단순 키워드 검색으로는 불가능하고, 의미적 연결이 필요합니다.

**3. proactive_recall (선제적 회상)** — 20문항
> "묻지 않아도 먼저 관련 정보를 제안하는가?"

예시: 사용자가 "서울 여행 계획 중이야"라고 말할 때, AI가 자발적으로 "아, 서울에 친구가 있다고 하셨죠?"라고 꺼내는 것. 수동적 응답이 아닌 능동적 제안 능력을 측정합니다.

**4. abstention (환각 방지)** — 20문항
> "모르는 것은 모른다고 말하는가?"

예시: 사용자가 혈액형을 알려준 적 없는데 "내 혈액형이 뭐야?"라고 물을 때. AI가 없는 정보를 지어내면 FAIL, "그런 정보가 없어요"라고 하면 PASS. 이 테스트를 하나라도 실패하면 전체 등급이 F입니다.

### 🟡 고급 메모리 능력

**5. irrelevant_isolation (개인정보 격리)** — 15문항
> "관련 없는 질문에 개인정보를 끼워 넣지 않는가?"

예시: "파이썬 리스트 정렬하는 방법은?"이라고 물을 때, AI가 "안녕하세요 홍길동님, 소프트웨어 엔지니어이시죠? 정렬 방법은..."처럼 불필요하게 개인정보를 포함하면 FAIL입니다.

**6. multi_fact_synthesis (다중 사실 통합)** — 20문항
> "여러 기억을 조합해 복합적인 답을 만들 수 있는가?"

예시: "내 예산, 취미, 일정을 고려해서 다음 주 활동을 추천해줘" → 직업·소득·취미·시간대 등 여러 사실을 동시에 연결해야 합니다.

**7. entity_disambiguation (동명이인 구분)** — 20문항
> "같은 이름의 다른 사람을 맥락으로 구분하는가?"

예시: 친구 "Alex(개발자)"와 형 "Alex(의사)"가 저장된 상황에서 "Alex한테 물어봤어"라고 했을 때, 어떤 Alex를 말하는지 맥락으로 파악하는 능력입니다.

**8. contradiction_direct (직접 모순 처리)** — 20문항
> "명확한 정보 변경을 정확히 업데이트하는가?"

예시: "나 이제 카페 알바 그만뒀어"라고 했을 때, 기존의 "카페 알바 중" 기억을 적절히 업데이트하는지 봅니다.

**9. contradiction_indirect (간접 모순 처리)** — 15문항
> "암시적인 모순을 추론으로 감지하는가?"

예시: "담배 끊은 지 3년 됐어"라고 했을 때, 이전에 저장된 "흡연자" 기억이 자동으로 업데이트되어야 합니다. 직접적인 수정 명령 없이 맥락으로 추론해야 합니다.

**10. noise_resilience (노이즈 저항성)** — 20문항
> "관련 없는 정보들 사이에서 올바른 기억을 찾는가?"

예시: 대화 중 여러 무관한 주제가 섞여 있을 때, 정작 필요한 기억을 정확히 꺼내는 능력입니다.

### 🔴 지속성·시간 능력

**11. unchanged_persistence (불변 기억 보존)** — 15문항
> "일부 정보 업데이트 후 다른 기억들은 그대로인가?"

예시: 직업을 업데이트한 후 "내 이름이 뭐야?" "취미는?"처럼 변경하지 않은 다른 사실들도 여전히 정확히 기억하는지 봅니다.

**12. temporal (시간적 기억)** — 25문항
> "현재 상태와 과거 상태를 구분하여 회상할 수 있는가?"

예시: "내가 예전에 무슨 일 했었어?"라고 물을 때, 현재 직업이 바뀌었더라도 과거의 직업 기록을 정확히 꺼내는 능력입니다.

### [Claude Sonnet의 분석]

R5 결과는 **기억력과 자기인식(self-awareness)의 근본적 트레이드오프**를 정량적으로 드러낸다.

**기술적 분석**: Naia는 3위(84.0%)로 준수하나, abstention(환각 방지, 45%)과 unchanged_persistence(불변 기억 보존, 47%)의 저조가 아킬레스건이다. 역설적으로 이 두 항목 실패는 같은 뿌리에서 발생한다 — 메모리가 풍부할수록 모르는 것도 "아는 척"하고, 업데이트 후 이전 기억과의 경계가 흐려진다. Graphiti는 Neo4j 기반 지식 그래프의 한계를 보여준다: contradiction 100%라는 구조적 강점이 있지만 semantic_search(의미 기반 검색) 4%는 그래프 탐색이 임베딩 기반 유사도 검색을 대체할 수 없음을 방증한다.

**설계 통찰**: Letta가 1위를 차지한 이유는 단일 지표 최적화가 아닌 균형에 있다. direct_recall(직접 회상) 96%와 proactive_recall(선제적 회상) 100%의 공존은 검색(reactive)과 추천(proactive) 경로를 분리 설계했음을 시사한다. 메모리 시스템의 성숙도는 "얼마나 많이 기억하는가"가 아니라 **"언제 모른다고 말하는가"** 로 측정된다.

**Naia 개선 방향**:
1. **Confidence-gated recall**: 검색 유사도 임계값 미달 시 응답 억제 로직 추가 (abstention(환각 방지) 복원)
2. **Immutable snapshot store**: 업데이트 전 구 사실을 `superseded` 태그로 보존, temporal(시간적 기억)/unchanged_persistence(불변 기억 보존) 동시 해결
3. **Dual-path retrieval**: 키워드 경로(direct_recall(직접 회상)) + 임베딩 경로(semantic) 병렬 실행 후 교차 검증

**독창적 관찰**:
- airi(no-memory)의 abstention(환각 방지) 100%는 블랙유머다 — 기억이 없으면 환각도 없다. 이는 메모리 시스템이 해결해야 할 문제가 "저장"이 아닌 **"경계 인식"** 임을 역설한다.
- 메모리 있는 모든 시스템의 abstention(환각 방지) 동반 하락(40~65%)은 개별 구현 버그가 아닌 **메모리-확신 간 구조적 결합** 문제다. 별도의 "불확실성 모델링" 레이어가 업계 표준이 되어야 할 시점이다.
- Naia의 noise_resilience(노이즈 저항성) 100%는 4-store 아키텍처의 잠재력을 보여준다. 복잡한 인지 구조가 노이즈 필터링에 실질적으로 기여하고 있다.
- SillyTavern의 temporal(시간적 기억) 96%는 설계 원칙의 역설을 보여준다. 롤플레이 챗봇용으로 설계된 시스템이 temporal(시간적 기억)에서 1위를 차지한 것은, "캐릭터의 과거 발언과 현재 상태의 연속성"을 유지하는 것이 롤플레이의 핵심이기 때문이다. 목적이 명확한 시스템이 범용 메모리보다 특정 능력에서 우월할 수 있다.
- multi_fact_synthesis(다중 사실 통합) 100%와 abstention(환각 방지) 저조는 서로 반대 방향의 능력이다 — 하나는 "연결할 수 있는 것을 연결하라", 다른 하나는 "없는 것을 만들지 말라". 이 둘을 모두 만족하는 시스템은 아직 존재하지 않는다. 이것이 AI 메모리 시스템의 본질적 이진 딜레마일 것이다.

*가장 중요한 단일 시사점: 메모리 시스템을 "더 많이 기억하도록" 최적화하는 것은 틀린 방향이다. 진짜 과제는 무엇을 기억하고, 무엇을 잊고, 무엇을 모른다고 말해야 하는지를 구분하는 능력이다.*

**"환각 방지의 역설" — AI 신뢰도의 본질**

이번 벤치마크에서 가장 흥미로운 발견은 "환각 방지의 역설"입니다. 메모리가 강한 시스템일수록 abstention(환각 방지) 점수가 낮습니다. letta 50%, naia 45%, mem0 40%... 반면 메모리 없는 airi는 100%입니다.

이것이 의미하는 바: 풍부한 기억은 자신감을 높이고, 과도한 자신감은 환각을 낳습니다. AI의 신뢰성은 "많이 아는 것"이 아니라 "자신이 모른다는 사실을 아는 것(메타인지)"에 달려 있습니다. 미래의 AI 메모리 시스템은 회상 능력뿐 아니라 **불확실성을 명시적으로 표현하는 능력**을 함께 갖춰야 합니다.

**독창적 관점**:
- 이 벤치마크는 "회상" 능력을 측정하지만, 진정으로 유용한 기억 시스템은 **성찰(reflection)** 능력 — 과거의 실수를 학습하는 능력 — 도 갖춰야 합니다. 다음 벤치마크에는 이 차원이 추가되어야 합니다.
- "기억의 비용"이 측정되지 않았습니다. 완벽한 기억은 공짜가 아닙니다 — 토큰 비용, 응답 지연, 저장 비용. 실용적 AI 시스템은 효율적인 망각을 통해 중요한 정보만 선별 유지해야 합니다.
- unchanged_persistence(불변 기억 보존)에서 airi가 100%를 기록한 것은 "변경한 것이 없으니 망가진 것도 없다"는 의미입니다. 이는 정적 시스템이 동적 시스템보다 특정 지표에서 유리한 구조적 편향입니다.

### 합의 1: "abstention(환각 방지) 역설"은 메모리 업계 전체의 미해결 과제

**Claude**: semantic_search(의미 기반 검색) 4%는 그래프 탐색과 임베딩 검색이 근본적으로 다른 기술임을 보여준다.

**Gemini**: 데이터 무결성·기업 환경에서는 graphiti가 오히려 적합하다. 다른 평가 기준이 필요한 시스템이다.

**GLM**: 그래프 DB와 벡터 DB는 상호 보완 관계다. 두 기술을 합친 하이브리드 아키텍처가 답이다.

**합의**: 단일 지표로 메모리 시스템을 비교하는 것에는 한계가 있다. "활용 목적별" 최적 시스템이 다르며, 이 벤치마크는 "AI 컴패니언" 사용 사례에 편향되어 있음을 인정해야 한다.

### 합의 3: Naia의 현 위치와 가야 할 방향

**Claude**: 84%는 좋은 시작이지만 abstention(환각 방지, 45%)과 unchanged_persistence(불변 기억 보존, 47%)의 동반 해결 없이는 실용화가 어렵다.

**Gemini**: noise_resilience(노이즈 저항성)·multi_fact_synthesis(다중 사실 통합) 100%는 경쟁력 있는 차별점이다. 이 강점을 유지하면서 약점을 보완해야 한다.

**GLM**: unchanged_persistence(불변 기억 보존)는 능력 한계가 아닌 버그(naia-os#221)다. 수정 후 재측정이 필요하다.

**합의**: Naia는 현재 "높은 잠재력을 가진 불완전한 시스템"이다. 1개의 버그 수정(#221)만으로도 순위가 크게 달라질 가능성이 있다.

## 7. Naia 개선 방향 상세 로드맵

### 단기 (1-2스프린트)

**[P0] naia-os#221 — unchanged_persistence(불변 기억 보존) 버그 수정**
- 원인: contradiction resolution 시 연관 엔티티 cascade 덮어쓰기
- 수정: 엔티티 단위 트랜잭션 격리, 수정 대상 속성만 업데이트
- 예상 효과: unchanged_persistence(불변 기억 보존) 47% → 85%+ 전망

**[P1] Confidence-gated recall — abstention(환각 방지) 개선**
- 검색 코사인 유사도 0.75 미만 시 "해당 정보 없음" 명시 반환
- 예상 효과: abstention(환각 방지) 45% → 70%+ 전망

### 중기 (1분기)

**[P2] Bi-temporal 모델 도입 — temporal(시간적 기억) 개선**
- 각 사실에 `valid_from`, `valid_until` 타임스탬프 추가
- 과거 상태 쿼리 가능: "내가 2년 전에 무슨 일 했었어?"
- 예상 효과: temporal(시간적 기억) 80% → 90%+ 전망

**[P3] Dual-path retrieval — direct_recall(직접 회상)/semantic 개선**
- 키워드 BM25 경로 + 임베딩 경로 병렬 실행 후 RRF(Reciprocal Rank Fusion) 통합
- 예상 효과: direct_recall(직접 회상) 84% → 92%+ 전망

### 장기 (2026 H2)

**[P4] 불확실성 명시 레이어**
- 모든 recall 응답에 신뢰도 점수 포함
- LLM 프롬프트에 "신뢰도 X% 미만 정보는 추측임을 명시" 지시 추가

**[P5] 메모리 효율 최적화**
- 에빙하우스 망각 곡선 기반 자동 압축·아카이빙

## 8. 결론

### 이번 벤치마크가 보여준 것

1. **AI 메모리는 이제 성숙기에 진입했다**: letta(87.5%)와 naia(84.0%)의 점수는 AI가 단순한 질의응답 도구를 넘어 진정한 "기억하는 AI"로 발전할 수 있음을 보여줍니다.

2. **완벽한 메모리 시스템은 아직 없다**: 모든 시스템이 F 등급을 받았습니다. 특히 모든 메모리 시스템이 환각 방지에 실패하는 패턴은 업계 전체가 해결해야 할 공통 과제임을 시사합니다.

3. **"무엇을 기억하는가"보다 "무엇을 모른다고 말하는가"가 더 중요하다**: abstention(환각 방지) 역설은 AI 신뢰성의 본질이 정보량이 아닌 메타인지 능력에 있음을 드러냅니다.

4. **아키텍처가 운명을 결정한다**: graphiti처럼 하나의 아키텍처 선택(Neo4j)이 모든 카테고리 점수에 일관된 영향을 미칩니다. 메모리 시스템 설계는 단순한 구현이 아닌 철학적 선택입니다.
